# Module 1 Changelog

## 2026-02-20 (v8.1) — ProbLog Backend Auto-Detection
- **Bug**: ProbLog's default `ddnnf` evaluation backend uses an external C binary (`dsharp`) that can segfault (SIGSEGV) if system libraries are incompatible with the pre-compiled binary shipped in the ProbLog package.
- **Fix**: Added `_detect_problog_backend()` function in `knowledge_base.py` that runs a single-fact probe query at import time. If `ddnnf`/dsharp works → use it (fast, ~2s for 44 tests). If it crashes → fall back to `kbest` (pure Python, ~273s for 44 tests) with a warning.
- **Portability**: No config needed. Anyone cloning the repo gets auto-detection. If their dsharp works, they get full speed. If not, they get a working fallback plus a log message explaining how to rebuild dsharp from source (https://github.com/QuMuLab/dsharp).
- **kbest tuple handling**: `kbest` backend returns `(probability, bound)` tuples instead of plain floats. Added tuple unpacking in result extraction: `float(val[0]) if isinstance(val, tuple) else float(val)`.
- **Root cause**: The `dsharp` binary at `.venv/.../problog/bin/linux/dsharp` is a pre-compiled C executable. System library updates (glibc/libstdc++) can break binary compatibility. Rebuilding from source fixes it: `git clone https://github.com/QuMuLab/dsharp && cp Makefile_gmp Makefile && make` then copy the binary to the problog package directory.

## 2026-02-20 (v8) — MusicBrainz Editorial Metadata Integration
- Integrated MusicBrainz editorial/structural metadata into the compatibility heuristic, completing the three-layer data architecture: content-based (AcousticBrainz), user-behavioral (ListenBrainz), editorial (MusicBrainz).
- **New file: `musicbrainz_helpers.py`** — separated to maintain clean data source distinction.
  - `artist_compatibility_prob`: Graph topology on artist relationships. Same artist → 0.95, 1-hop relationship (member-of, collaboration, producer) → 0.70, no relationship → 0.50 (neutral). Research: Korzeniowski et al. (ISMIR 2021) showed graph topology alone outperforms content features for artist similarity (NDCG@200 = 0.45 vs 0.24).
  - `era_compatibility_prob`: Gaussian decay on release year difference (σ=5.0 years). Same year → 100%, 5yr → 61%, 10yr → 14%, 20yr → <1%. Research: Schweiger et al. (EPJ Data Science 2025) showed temporal proximity is key to playlist coherence.
  - `mb_genre_compatibility_prob`: Jaccard similarity on MusicBrainz curated genre taxonomy. Case-insensitive. Research: Serra & Alonso (ISMIR 2022) showed editorial metadata outperforms user-generated tags for music representation.
- **New fields in `TrackFeatures`**: `mb_artist_related_mbids` (set of related artist MBIDs), `mb_release_year` (first release year), `mb_genre_tags` (curated genre list).
- **New fields in `UserPreferences`**: `artist_weight` (default 0.10), `era_weight` (default 0.05), `mb_genre_weight` (default 0.0 — LB tags already carry genre signal).
- **New fields in `TransitionResult`**: `artist_compatibility`, `era_compatibility`, `mb_genre_compatibility`.
- **`music_theory.pl` v8**: Added default clauses for `artist_compatible`, `era_compatible`, `mb_genre_compatible`. Updated `smooth_transition` composite rule.
- **21 new tests** in `test_musicbrainz.py`: artist compatibility (5 tests), era compatibility (6 tests), MB genre compatibility (6 tests), knowledge base integration (4 tests covering same-artist boost, era scoring, neutral fallback, weight effect).
- **Total scoring dimensions**: 12 (7 content-based + 2 user-behavioral + 3 editorial). All auto-normalized.
- **Decision — Why only 3 MusicBrainz dimensions?** MusicBrainz also provides recording relationships (covers, remixes, samples), work-level data (same composition by different performers), label, country, instrument credits, and series membership. We selected the 3 dimensions with the strongest research backing and broadest data coverage: (1) artist relationships — Korzeniowski showed graph topology alone outperforms content features; (2) era/release year — Schweiger showed temporal proximity is key to playlist coherence; (3) curated genre taxonomy — Serra showed editorial metadata outperforms user tags. The remaining dimensions have diminishing returns: recording relationships are sparse (most tracks aren't covers/remixes of each other), work-level data is mostly relevant to classical repertoire, and label is noisy (major labels too broad, niche labels too sparse). These could be added later if needed.
- **Decision — Weight rationale**: Default weights are informed by the literature but not empirically optimized against ground-truth playlist data. The ratios reflect relative importance from research: tempo (0.20) is highest because BPM mismatch is the most jarring transition artifact (Drake & Botte 1993). Key, energy, mood, timbre (all 0.15) are next — Aucouturier & Pachet (2002) showed MFCCs (timbre) are the dominant content similarity signal; key and mood are perceptually salient. Artist (0.10) and tags (0.10) are strong structural signals but not acoustic. Genre (0.05), loudness (0.05), and era (0.05) are supporting signals. Popularity (0.0) and MB genre (0.0) are off by default — popularity to stay discovery-friendly, MB genre because LB tags already carry the genre signal. Auto-normalization (dividing by sum of active weights) means users can adjust any weight without rebalancing the rest. Proper calibration would require optimizing weights against a dataset like the Million Playlist Dataset to maximize playlist coherence — this is deferred.
- **Decision — Dimension evolution (raw features → 12 scoring dimensions)**:
  - **Available raw features (~70+)**: AcousticBrainz lowlevel JSON alone contains 60-80+ feature groups: 13 MFCCs + 13x13 covariance matrix, 28 bark bands, 40 mel bands, 4 spectral energy bands, spectral centroid/flux/complexity/rolloff/contrast/entropy/kurtosis/skewness, pitch salience, zero crossing rate, hfc, gfcc, dissonance, loudness, dynamic complexity, BPM, danceability, onset rate, etc. AcousticBrainz highlevel adds 6 mood classifiers, timbre, genre (8-class distribution), voice/instrumental, gender, tonal/atonal. On top of that: ListenBrainz provides user tags + listen counts, MusicBrainz provides artist relationships + release year + curated genres + recording relationships + work-level data + label + country + instruments.
  - **Original plan (v0)**: Proposed 6 scoring dimensions — key, tempo, energy, loudness, mood, timbre. Used hand-tuned threshold rules in ProbLog (e.g., "< 5 BPM = 0.95, 5-10 BPM = 0.85"). Many raw features were extracted into `TrackFeatures` but never used in scoring (onset_rate, beats_count, tuning_frequency, chords_strength, spectral_centroid, dissonance, dynamic_complexity, danceability).
  - **v1-v6 refinement (6 → 7 dimensions)**: Added genre as 7th dimension. Replaced all hand-tuned thresholds with research-grounded continuous functions (Krumhansl-Kessler key profiles, Weber's law tempo, Bhattacharyya timbre distance). Key insight: continuous probability functions eliminated the need for arbitrary threshold bins and produce smoother gradient signals for beam search.
  - **v7 ListenBrainz (7 → 9 dimensions)**: Added tags (cosine similarity) and popularity (log-Gaussian). Evaluated GMM on tag space — rejected as cosine similarity captures ~90% of the signal for far less complexity. Reduced timbre weight 20%→15% and genre weight 10%→5% since LB tags carry richer genre signal than 8-class rosamerica.
  - **v8 MusicBrainz (9 → 12 dimensions)**: Added artist (graph topology), era (Gaussian decay on year), MB genre (Jaccard). Selected based on 3 papers showing these are the highest-impact editorial signals.
  - **What's good about 12**: (1) Each dimension captures a distinct, orthogonal signal — no redundancy between key/tempo/timbre (acoustic) vs tags/popularity (behavioral) vs artist/era/genre (editorial). (2) Missing data falls back to 0.5 (neutral) on any dimension — the system degrades gracefully when only AcousticBrainz data is available, or only MusicBrainz, etc. (3) Auto-normalized weights mean the system works with any subset of data. (4) The unused raw features (onset_rate, spectral_centroid, bark bands, mel bands, etc.) aren't wasted — they're either captured indirectly (mel bands → timbre via MFCC), too noisy to help (spectral_centroid alone), or better suited as Module 2/3 constraints (danceability, dynamic_complexity).
  - **Known limitations**: (1) Energy uses spectral energy bands, not perceptual energy — a quiet acoustic guitar and a quiet string quartet may score similarly despite different perceived energy. (2) Energy/loudness Gaussian decay is symmetric — doesn't capture directional energy flow (building up vs winding down). The `prefer_increasing_energy`/`prefer_decreasing_energy` fields exist in UserPreferences but are reserved for Module 2/3 as search constraints. (3) σ values for energy (0.003) and loudness (0.15) were calibrated to a narrow test set (4 tracks). Should be validated against a larger corpus.
- **Decision — AcousticBrainz dependency risk and mitigation**:
  - **The problem**: AcousticBrainz was shut down in 2022 (no new submissions accepted). 7 out of 12 scoring dimensions depend on AcousticBrainz data: key, tempo, energy, loudness, mood, timbre, and genre (rosamerica). These are the core content-based features — without them, only 5 dimensions remain active: tags and popularity (ListenBrainz, still active), artist, era, and MB genre (MusicBrainz, still active). Of those 5, popularity and MB genre are off by default, leaving effectively 3 active dimensions (tags, artist, era) for tracks without AB data.
  - **MusicBrainz does NOT replace AcousticBrainz**: MusicBrainz is purely editorial/structural metadata (artist relationships, release dates, genre taxonomy, credits). It has zero audio-derived features — no MFCCs, no spectral analysis, no mood classifiers, no BPM detection. The two databases serve fundamentally different purposes despite being in the same MetaBrainz ecosystem.
  - **Mitigation strategies**: (1) **AcousticBrainz data dump** (available at acousticbrainz.org/download): The full dataset was archived before shutdown. For any track that was submitted before 2022, the lowlevel/highlevel JSON is still available. This covers a large catalog of existing music but no new releases. (2) **Essentia** (essentia.upf.edu): The open-source audio analysis library developed by MTG Barcelona — the same tool AcousticBrainz used internally to extract features. Still actively maintained. Can extract identical features (MFCCs, spectral energy, BPM, key) from raw audio files. Requires access to audio files, which Module 2 would need to handle. (3) **Spotify Audio Features API**: Was a convenient alternative but Spotify deprecated/restricted this endpoint in late 2024 for new developer apps. Not a reliable long-term option.
  - **Sustainable path forward**: Use the AcousticBrainz data dump for tracks already in the catalog (covers most pre-2022 music). For new releases and tracks not in the dump, use Essentia to extract features directly from audio files. This keeps the same feature space (MFCCs, spectral bands, BPM, key estimation) with no model changes needed — only the data pipeline in Module 2 changes. The scoring functions in Module 1 are agnostic to whether features came from AcousticBrainz or Essentia.
  - **Current graceful degradation**: The system already handles missing AB data correctly — all 7 content-based dimensions fall back to 0.5 (neutral) when data is absent, and weight normalization ensures the remaining active dimensions still produce meaningful scores. This means the system works today with partial data, just with reduced signal quality.

## 2026-02-19 (v7) — ListenBrainz Integration
- Integrated ListenBrainz user-behavioral signals into the compatibility heuristic, alongside existing AcousticBrainz content-based features.
- **New file: `listenbrainz_helpers.py`** — separated from `rules_helpers.py` to maintain clean acoustic vs user-behavioral distinction.
  - `tag_compatibility_prob`: Cosine similarity on ListenBrainz user-generated tag count vectors (e.g. `{"shoegaze": 45, "dreampop": 32}`). Returns [0, 1], 0.5 for missing data. Far richer than rosamerica 8-class genre classifier.
  - `popularity_compatibility_prob`: Gaussian decay on log-scale listen count difference (sigma=2.0). Same order of magnitude ~100%, 10x diff ~52%, 1000x diff <1%. Returns 0.5 for missing data.
- **New fields in `TrackFeatures`**: `tags`, `artist_mbid`, `popularity_listen_count`, `popularity_user_count` — populated by Module 2's ListenBrainz client.
- **New fields in `UserPreferences`**: `tag_weight` (default 0.10), `popularity_weight` (default 0.0 — discovery-friendly), `discovery_mode` (default False).
- **New fields in `TransitionResult`**: `tag_compatibility`, `popularity_compatibility`.
- **Weight normalization**: Replaced flat weighted sum with normalized weighted average. Weights auto-normalize so users can adjust tag/popularity weights without rebalancing everything else.
- **Discovery mode**: When `discovery_mode=True`, popularity weight is forced to 0.0 in the heuristic. Module 2 will use this flag to widen ListenBrainz candidate generation (mode=hard, low pop_begin).
- **Reweighted defaults**: timbre 20%→15%, genre 10%→5% (tag dimension now carries the richer genre signal).
- **`music_theory.pl` v7**: Added default clauses for `tag_compatible`, `has_tag_data`, `popularity_compatible`, `has_popularity_data`. Updated `smooth_transition` composite rule.
- **17 new tests** in `test_listenbrainz.py`: tag compatibility (6 tests), popularity compatibility (6 tests), knowledge base integration (5 tests covering tags flow-through, no-tags fallback, discovery mode, weight normalization, tag weight effect).
- **Decision**: Evaluated GMM on tag space — cosine similarity already captures ~90% of the signal for the heuristic's purpose. GMM adds value only for zero-overlap/same-genre edge cases, not worth the pipeline complexity (training corpus, dimensionality reduction, sklearn dependency). Keeping cosine similarity.

## 2026-02-10 (v6)
- Replaced all hand-tuned threshold rules with research-grounded continuous probability functions:
  - **Key**: Krumhansl-Kessler (1990) probe-tone profiles + Pearson correlation. Captures relative major/minor and tritone nuances that circle-of-fifths distance misses.
  - **Tempo**: Weber's law Gaussian decay (Drake & Botte 1993). JND ≈ 4%, σ = 3×JND = 0.12. Handles double/half-time via min(Δ_direct, Δ_double, Δ_half).
  - **Timbre**: Bhattacharyya coefficient BC = exp(-D_B) (Aucouturier & Pachet 2002). Principled distance-to-probability conversion.
  - **Loudness**: Gaussian decay σ=0.15 on AcousticBrainz 0-1 scale.
  - **Energy**: Gaussian decay σ=0.003 calibrated to spectral energy band range.
- All 5 components now computed in Python and asserted as probabilistic facts (same pattern as mood/genre from v5).
- `music_theory.pl` reduced to: default clauses + mood noisy-OR + genre dot product + composite rules + preference rules. No threshold rules remain.

## 2026-02-10 (v5)
- Replaced deterministic mood/genre facts with proper ProbLog probabilistic inference:
  - **Mood**: Independent probabilistic facts `P::mood_happy(T)`, `P::mood_sad(T)`, etc. from AcousticBrainz classifier probabilities. ProbLog computes noisy-OR: P(mood_compatible) = 1 - ∏(1 - P(mood_X(T1)) × P(mood_X(T2))) over 6 mood dimensions. No hand-tuned cross-mood bonuses.
  - **Genre**: Annotated disjunctions from full 8-class rosamerica distribution. ProbLog computes dot product: P(genre_compatible) = Σ_g P(genre(T1,g)) × P(genre(T2,g)). Pop vs classical → 5%, classical vs classical → 56%.
- Removed `_get_dominant_mood()` (collapsed distribution to argmax), replaced with `_get_top_mood()` (display only).
- Added `has_mood_data`/`has_genre_data` guard predicates with 50% fallback for missing data.
- Note: ProbLog `Term.__init__` does NOT convert `p=float` to `Constant`; must use `p=Constant(prob)` explicitly.

## 2026-02-10 (v3)
- Replaced Euclidean MFCC distance with Bhattacharyya distance using full covariance matrices from AcousticBrainz; models each song as single Gaussian N(μ,Σ) — captures timbral spread, not just mean (Aucouturier & Pachet 2002).
- Recalibrated ProbLog timbre thresholds for Bhattacharyya scale: <0.5 (very similar), 0.5–1.0 (similar), 1.0–2.0 (cross-genre), ≥2.0 (very different).
